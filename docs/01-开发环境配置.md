### 2.0 本仓库目录说明（先看）

| 路径 | 说明 |

**`vendor` 命名卷：** `zhixin-vendor` 挂载到 `/var/www/html/vendor`，避免 Windows 绑定挂载数万小文件导致 **整站极慢**。该卷**首次为空**；现已在 **`docker-entrypoint.sh`** 中于 **`php-fpm` 启动前**自动执行 `composer install`。若自动安装失败（网络、镜像无 Composer 等），页面会出现 **`Failed opening required ... vendor/autoload.php`**，请按上表第 4 步手动执行，或使用：

## 🐳 三、Docker 配置详解（补充）

### 3.1 docker-compose.yml（与仓库一致摘要）

> Compose 文件顶部的 `version` 字段在新版 Docker 中可省略；若出现 *obsolete* 提示可忽略或删除该行。

```yaml
# 文件位置：_laravel_temp/docker-compose.yml（挂载「当前目录 .」即 Laravel 根）
services:
  php:
    build:
      context: ./docker/php
      dockerfile: Dockerfile
    working_dir: /var/www/html
    volumes:
      - .:/var/www/html
      - zhixin-vendor:/var/www/html/vendor
    depends_on:
      - mysql
      - redis

  nginx:
    image: nginx:alpine
    ports:
      - "8083:80"
    volumes:
      - .:/var/www/html
      - ./docker/nginx/conf.d:/etc/nginx/conf.d
    depends_on:
      - php

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: aiside123456
      MYSQL_DATABASE: openclaw_zhixin
      MYSQL_USER: ai_side
      MYSQL_PASSWORD: aiside123456
    volumes:
      - mysql-data:/var/lib/mysql
    ports:
      - "3309:3306"

  redis:
    image: redis:7.0-alpine
    volumes:
      - redis-data:/data

volumes:
  mysql-data:
  redis-data:
  zhixin-vendor:
```

### 3.2 Dockerfile（PHP，与 `_laravel_temp/docker/php/Dockerfile` 一致）

```dockerfile
FROM php:8.2-fpm

COPY --from=composer:2 /usr/bin/composer /usr/bin/composer

RUN apt-get update && apt-get install -y --no-install-recommends \
    git unzip libzip-dev libpng-dev libonig-dev libxml2-dev $PHPIZE_DEPS \
    && docker-php-ext-install pdo_mysql mbstring zip gd \
    && pecl install redis \
    && docker-php-ext-enable redis \
    && rm -rf /var/lib/apt/lists/*

COPY docker-entrypoint.sh /usr/local/bin/openclaw-php-entrypoint.sh
RUN chmod +x /usr/local/bin/openclaw-php-entrypoint.sh

WORKDIR /var/www/html

ENTRYPOINT ["/usr/local/bin/openclaw-php-entrypoint.sh"]
CMD ["php-fpm"]
```

**`docker-entrypoint.sh` 行为：** 若 `vendor/autoload.php` 不存在，先执行 `composer install`（若首次失败会自动加 `--ignore-platform-reqs` 重试），再启动 `php-fpm`。镜像构建阶段已在 `Dockerfile` 中配置 `git safe.directory`，避免挂载目录触发 *dubious ownership*；**宿主机上** `git pull` 若仍报该错误，请用仓库所有者用户执行，或执行一次：`git config --global --add safe.directory /你的项目绝对路径`。

## 🐛 五、常见问题

### 5.1 容器启动失败

**问题：** `Error: Cannot create container for service php`

**解决：**
```bash
# 1. 删除旧容器
docker-compose down

# 2. 重新构建
docker-compose build --no-cache

# 3. 重新启动
docker-compose up -d
```

### 5.2 数据库连接失败

**问题：** `SQLSTATE[HY000] [2002] Connection refused`

**解决：**
```bash
# 1. 检查 MySQL 容器状态
docker-compose ps mysql

# 2. 查看 MySQL 日志
docker-compose logs mysql

# 3. 确保 MySQL 已启动
docker-compose up -d mysql

# 4. 等待 30 秒后重试
sleep 30
docker-compose exec php php artisan migrate
```

### 5.3 权限问题

**问题：** `Permission denied`

**解决：**
```bash
# 1. 修复文件权限
docker-compose exec php chown -R www-data:www-data /var/www/html

# 2. 修复存储目录权限
docker-compose exec php chmod -R 775 storage bootstrap/cache

# 3. 重启容器
docker-compose restart php
```

**关联现象：** 注册页「获取验证码」报 **`storage/logs/laravel.log` Permission denied** 并返回 500 时，除上述权限修复外，项目已默认采用 **stderr + 单文件** 日志栈且 **忽略子通道异常**（见 `_laravel_temp/config/logging.php`），避免因日志落盘失败阻断接口。

### 5.3.1 注册验证码「显示已发送但收不到」

- **行为说明：** 仅当 SMTP **真正发信成功** 后才会写入验证码与 60 秒频控；失败时接口返回 **422**，前端展示错误文案（不再误报「已发送」）。
- **QQ 邮箱：** `smtp.qq.com`、`465`、`MAIL_ENCRYPTION=ssl`、密码为 **SMTP 授权码**（非 QQ 登录密码）；**`MAIL_FROM_ADDRESS` 建议与 `MAIL_USERNAME` 相同**。
- **耗时：** 已设默认 **`MAIL_TIMEOUT=30`**（秒），避免长时间阻塞；生产可配合队列异步发信（后续优化）。

### 5.4 `vendor/autoload.php` 不存在 / 白屏 Fatal error

**现象：** `Failed opening required '/var/www/html/vendor/autoload.php'`

**原因：** 命名卷 `zhixin-vendor` 挂载后**覆盖**了容器内 `vendor`，新卷为空；若启动时 **composer install** 失败或未执行，则缺少依赖。

**处理：**

```bash
docker compose exec php composer install --no-interaction
docker compose restart php
```

或重建 PHP 镜像（含最新 `docker-entrypoint.sh`）后：`docker compose build php && docker compose up -d`，查看 `docker compose logs php` 是否自动安装成功。

### 5.5 Laravel 安装失败

**问题：** `composer create-project 失败`

**解决：**
```bash
# 1. 检查 Composer 版本
docker-compose exec php composer --version

# 2. 更新 Composer
docker-compose exec php composer self-update

# 3. 清理 Composer 缓存
docker-compose exec php composer clear-cache

# 4. 重新安装
docker-compose exec php composer create-project laravel/laravel:^10.0 .
```

## 🌐 九、生产上线要点（与本地 Docker 的差异）

### 9.0 【服务器/Linux】部署路径与 `docker-compose`（与本地一致）

`docker-compose.yml` / `docker-compose.server.yml` 与 `docker/` 已放在 **`_laravel_temp/`**（Laravel 根）。全程使用**相对路径**（`./docker/php`、挂载 **`.` → `/var/www/html`**），**不需要**写绝对路径。只要在**存放 `docker-compose.yml` 的目录**下执行 Compose 即可。

**本地仓库结构：**

```text
openclaw_zhixin/
├── docs/
└── _laravel_temp/                   ← 在此目录执行 docker compose
    ├── docker-compose.yml
    ├── docker-compose.server.yml
    ├── docker/
    ├── artisan
    └── ...
```

**服务器：** 将 **`_laravel_temp` 目录内全部内容**同步到目标目录（例如 `/opt/zixin/openclaw-zixin`），使 **`artisan`、`docker/`、`docker-compose.server.yml` 同层**，然后：

```bash
cd /opt/zixin/openclaw-zixin
docker compose -f docker-compose.server.yml up -d --build
```

生产使用 **`docker-compose.server.yml`**（**`80:80`**）；本地开发使用同目录 **`docker-compose.yml`**（**`8083:80`**）。配置 `.env` 中 `APP_URL`。若宿主机 **80 已被占用**，需释放端口或改用反代。

**常见错误：**

| 现象 | 原因 |
|------|------|
| PHP/Nginx 白屏或找不到 `artisan` | 未在 Laravel 根目录执行 compose，或同步代码不完整 |
| 开发环境访问 `http://IP/` 无站 | 本地默认 **`8083`**，应访问 **`http://IP:8083/`**，或改用 `docker-compose.server.yml` |
| 每次 `build` 像「重新下镜像」 | 首次构建或 Dockerfile 变更会拉层/跑 `apt-get`，属正常 |

`docker/` 一般不用改；按需调整 `docker/nginx/conf.d/*.conf`。

---

本地 `docker-compose.yml` 用于开发；上线时通常**不照搬** compose，但下列 **Laravel 环境变量与能力**需在生产侧对齐：

| 项 | 开发（本仓库 Docker） | 生产建议 |
|----|------------------------|----------|
| **`APP_ENV` / `APP_DEBUG`** | `local` / `true` | `production` / **`false`** |
| **`APP_URL`** | `http://localhost:8083` | 真实 HTTPS 域名 |
| **`SESSION_DRIVER`** | `redis`（推荐） | `redis` 或托管会话；**勿**在慢盘/网络盘上长期用 `file` |
| **`REDIS_*`** | 指向 `redis` 服务名 | 云 Redis 内网地址、密码、TLS 等 |
| **PHP Redis** | 镜像内 **phpredis** 扩展 | 生产 PHP 同样需扩展 **或** 使用 **Predis** 包 |
| **`SESSION_SECURE_COOKIE`** | 本地 HTTP 可 `false` | HTTPS 下 **`true`** |
| **数据库** | `mysql` 容器 | 托管 RDS 等；**勿**提交真实 `.env` |
| **静态资源** | `PUBLIC_ASSET_FILEMTIME` / `ASSET_VERSION` | 部署流水线 bump 版本或 CDN |
| **演示支付** | `PAYMENT_SIMULATE=true` | **务必关闭**并对接真实支付 |

### 9.3 上传尺寸限制（Nginx / PHP）

- **本地 Docker（已内置）**
  - Nginx：在 `docker/nginx/conf.d/default.conf` 中设置：

    ```nginx
    client_max_body_size 10m;  # 如需更大可调成 20m/50m
    ```

  - PHP：建议在容器的 `php.ini` 或单独 `uploads.ini` 中同步：

    ```ini
    upload_max_filesize = 20M
    post_max_size = 20M
    ```

  - 修改后执行：

    ```bash
    docker compose restart nginx
    docker compose restart php
    ```

- **生产环境（需手工对齐）**
  - 在生产 Nginx 站点配置的 `server { ... }` 内同样增加：

    ```nginx
    client_max_body_size 10m;  # 或按业务需要调大
    ```

  - 在生产 PHP（php-fpm）对应的 `php.ini` 中，将：

    ```ini
    upload_max_filesize = 20M
    post_max_size = 20M
    ```

    调整为与 Nginx 一致或略大。
  - 然后分别重载：

    ```bash
    nginx -t && nginx -s reload          # 或 systemctl reload nginx
    systemctl restart php-fpm            # 或重启对应 PHP 服务/容器
    ```

  - 若线上出现 `413 Request Entity Too Large`，优先检查上述两个配置是否已按本节说明修改并重启。

**说明：** 生产部署步骤（Nginx 配置、进程守护、队列 Worker、日志采集）因平台而异，可单独维护《部署手册》；**不必**把服务器密码写进 `01-开发环境配置.md`，本文档以**本地与变量语义**为主。

### 9.1 生产上线可否参照本文流程？

可以参照，但要注意这是**开发环境优先**文档。生产上线建议采用下面“同流程、不同参数”的做法：

```bash
# 1) 拉代码（生产机）
git pull

# 2) PHP 依赖（生产）
composer install --no-dev --optimize-autoloader

# 3) 前端资源（生产）
npm ci
npm run build

# 4) 数据库迁移（谨慎，先备份）
php artisan migrate --force

# 5) 缓存与优化
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan event:cache

# 6) 队列与计划任务
php artisan queue:restart
# cron: * * * * * php /path/to/artisan schedule:run >> /dev/null 2>&1
```

### 9.2 与本地流程的关键差异（必须）

- **依赖安装参数不同**：生产必须 `composer install --no-dev`，避免安装开发包。
- **Vite 命令不同**：生产只跑 `npm run build`，不使用 `npm run dev` 常驻服务。
- **迁移命令不同**：生产需 `php artisan migrate --force`，并先做数据库备份。
- **缓存策略不同**：生产使用 `config:cache/route:cache`，本地开发通常 `optimize:clear`。
- **进程守护不同**：生产要有 `php-fpm + nginx + queue worker + cron` 守护，不能只靠单终端命令。
- **配置安全不同**：`APP_DEBUG=false`、真实域名 HTTPS、密钥/SMTP/Redis 使用安全注入，不落库明文。
