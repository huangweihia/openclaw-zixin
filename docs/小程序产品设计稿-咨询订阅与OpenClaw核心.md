# OpenClaw 智信 · 微信小程序产品设计稿（核心业务对齐版）

> **文档性质**：产品级设计说明（非接口规格书）。  
> **修订日期**：2026-04-11  
> **适用仓库**：`openclaw-zixin`（站点与 API）+ `openclaw-zixin-xcx`（小程序）

---

## 1. 商业定位与核心价值

### 1.1 一句话定位

**向企业与个人售卖「可订阅的资讯与咨询价值」**：用户为自己关心的主题付费订阅，系统通过 **OpenClaw** 按约定频率抓取、整理并**投递**到用户侧（站内内容、邮件/推送等，具体以站点配置为准）。

### 1.2 核心价值主张（对外话术）

| 维度 | 说明 |
|------|------|
| **订阅** | 用户声明兴趣（关键词、来源、排除词、频率等），对应 **SVIP 内容订阅**（`SvipSubscription`） |
| **获取** | OpenClaw 定时任务拉取待处理订阅列表（`GET /api/svip/subscriptions/list`），抓取后再回传（`POST /api/svip/subscriptions/data`） |
| **沉淀** | 抓取结果写入站点内容库（如文章等），形成可复用的数字资产 |
| **触达** | 按订阅配置的 `push_methods` 等触发推送；另有 **邮件订阅主题**（`EmailSubscription`，需 VIP/SVIP） |

### 1.3 与「卖咨询」的关系

- **咨询**在系统中体现为：**可配置的、持续交付的情报服务**（非单次聊天窗口），通过 **订阅条款 + OpenClaw 履约 + 会员权益** 打包销售。  
- **站点前台**承担：品牌、内容种草、会员与支付、订阅管理（后台/Filament）。  
- **小程序**承担：移动侧 **触达、阅读、账户与订阅状态感知、轻管理**，必须与上述闭环一致，不能做成脱离订阅业务的「纯资讯 App」。

## 2. 现有网站与后端能力地图（事实基线）

以下基于当前代码与路由，作为小程序设计的**约束与可复用能力**。

### 2.1 OpenClaw 与内容管线

| 能力 | 说明 | 代表接口 / 模块 |
|------|------|-----------------|
| 通用采集入库 | OpenClaw 向站点推送多类型内容 | `POST /api/openclaw/data`（`OpenClawDataController`） |
| 任务可观测 | 失败/超时等日志 | `openclaw_task_logs`、Filament「OpenClaw 日志」 |
| SVIP 专属内容包 | 供 OpenClaw 侧拉取 VIP 侧内容 | `GET /api/svip/content`（`SvipContentController`，Token 保护） |

### 2.2 SVIP「咨询式」订阅（核心业务实体）

| 字段维度 | 含义 |
|----------|------|
| `keywords` / `exclude_keywords` / `sources` | 用户想跟踪什么、不要什么、从哪些源 |
| `frequency` | 更新节奏 |
| `push_methods` | 结果如何触达用户 |
| `last_fetch_at` / `last_fetch_count` | 履约与结算侧可展示的运行数据 |

OpenClaw：**取列表 → 抓数据 → 回传 `subscriptions/data` → 更新订阅状态并触发推送**（见 `SvipSubscriptionController::pushData`）。

### 2.3 会员、邮件与公开内容

- **邮件订阅主题**：`POST /api/email-subscriptions`（需登录且 **VIP/SVIP/管理员**），`subscribed_to` 等主题与 `EmailSubscription::TOPICS` 对齐。  
- **公开内容**：`GET /api/public/*`（文章、分类、browse 下项目/案例/工具/SOP）——用于**获客、信任与试用**，与订阅业务形成漏斗。  
- **微信登录与账号统一**：`POST /api/wechat/mini/login`、`/wechat/mini/me` 等，保证小程序用户与站点用户同源，便于展示**同一套会员与订单**。

## 3. 小程序产品目标（对齐大方向）

1. **订阅可感知**：用户打开小程序能看懂「我订了什么、最近一次何时更新、下一步何时来」。  
2. **履约可阅读**：与订阅相关的新增内容在小程序内**可读、可检索入口**（在 API 允许前提下）。  
3. **转化可衔接**：公开内容 → 登录 → 会员/订阅说明 → **回站点完成复杂配置/支付**（小程序内可逐步加深，见阶段规划）。  
4. **品牌一致**：视觉主题（皮肤）、文案口径与站点「内容 · 会员 · 运营」一致。

## 4. 信息架构（IA）建议

### 4.1 底部导航（保持四 Tab，与现状兼容）

| Tab | 产品职责 | 与核心业务关系 |
|-----|----------|------------------|
| **首页** | 价值主张、最新/热门内容、快捷进入「文章 / 发现 / SBTI」 | 漏斗顶端 + 活跃度 |
| **发现** | 项目 / 案例 / 工具 / SOP / 文章 / 测评入口 | 展示站点公开库，支撑咨询话题种草 |

| **我的** | 账号、会员与积分、订单、收藏、帖子、**订阅相关摘要** | **订阅与付费用户的「控制台」入口** |

## 5. 功能清单（按优先级）

### 5.1 P0 — 已具备或与「内容 + 账号」强相关（维持）

| 功能点 | 说明 | 业务意义 |
|--------|------|----------|
| 微信登录 / 退出 | 与站点用户绑定 | 订阅与订单同源 |
| 个人资料与主题换肤 | 与 `skins` API 同步 | 品牌与会员感 |
| 文章列表 / 详情 / 分类 / 热门 | `public/articles` | 阅读与 VIP 内容曝光 |
| 发现 → 浏览列表与详情 | `public/browse/*` | 公域种草 |
| SBTI 人格测试 | `personality-quiz` | 运营与线索 |
| 我的：订单、收藏、帖子 | `wechat/mini/*` | 交易与 UGC 闭环 |
| 绑定邮箱 | 接收通知、跨端登录 | 与邮件订阅、重要触达衔接 |

### 5.2 P1 — 强烈建议补齐（直接服务「卖咨询 + 订阅」）

| 功能点 | 用户价值 | 依赖/说明 |
|--------|----------|-----------|
| **我的订阅（咨询订阅概览）** | 展示当前用户 `SvipSubscription` 列表：名称、关键词摘要、频率、上次抓取时间、是否启用 | 需新增 **用户态 API**（Sanctum）：读自己的订阅；**禁止**复用 OpenClaw 用的 Token 接口直接给前端 |
| **订阅更新动态** | 时间线：与本人订阅相关的「新文章/新入库」摘要（可先做只读列表） | 需后端聚合查询或专用视图表 |
| **通知触达说明 + 跳转** | 说明邮件/站内/若后续接微信订阅消息的规则；已绑定邮箱引导查收 | 与 `push_methods`、邮件订阅文案一致 |
| **会员 / SVIP 权益说明页（小程序内）** | 明确「订阅咨询包」包含什么、与公开文章区别 | 文案与站点价格页对齐，避免承诺过度 |
| **支付与下单（可选路径）** | 若咨询产品在站点完成支付：小程序展示订单状态即可；若需小程序内购买，需 **微信支付 + 后端订单对接** | 单独立项 |

### 5.3 P2 — 增强体验

| 功能点 | 说明 |
|--------|------|
| 邮件订阅主题管理 | 对接 `POST /api/email-subscriptions`（VIP 限制） |
| 消息中心 | 站内通知若已有表与 API，可接「我的」红点 |
| 客服 / 企业微信 | 商务咨询入口，与「卖咨询」一致 |
| 搜索增强 | 全站搜索若 H5 已有，可复用或简化 |

### 5.4 刻意不做或慎做（非目标）

| 项目 | 原因 |
|------|------|
| 在小程序内用 web-view 嵌整站后台 | 体验差、审核风险；配置类复杂操作仍建议 **H5/PC** |
| 把 OpenClaw **服务端 Token 接口**直接暴露给小程序 | 安全风险；用户态与机器态接口必须分离 |
| 承诺「实时」推送 | 实际依赖定时任务与微信能力，产品文案需写「按频率/尽力送达」 |

## 6. 核心用户旅程（简版）

### 6.1 新访客（未登录）

发现公开内容（首页/发现/文章）→ 理解价值 → 引导登录 → 绑定邮箱（可选）→ **去站点了解订阅套餐并完成付费**（或后续小程序支付）。

### 6.2 已付费订阅用户（咨询交付对象）

打开小程序 → **我的订阅** 看状态与上次更新 → 阅读「订阅相关新内容」→ 订单与发票（若站点支持）→ 需要改关键词/加来源 → **跳转站点或联系顾问**（视运营策略）。

### 6.3 运营 / 顾问侧（不在小程序内完成）

在 Filament 配置 SVIP 订阅、查看 OpenClaw 日志、处理异常——与小程序**数据同源展示**即可，不要求小程序替代后台。

## 7. 与 OpenClaw / 后端的接口分层原则（给研发）

| 调用方 | 接口类型 | 原则 |
|--------|----------|------|
| OpenClaw 定时任务 | `svip/subscriptions/list`、`svip/subscriptions/data`、`openclaw/data`、`svip/content` 等 | **机器 Token**，仅服务器持有 |
| 微信小程序 | `public/*`、`wechat/mini/*`、未来 `user/svip-subscriptions` 等 | **用户 Bearer（Sanctum）**，仅访问当前用户数据 |
| 管理后台 | Filament / 管理 API | 权限与审计 |

## 11. 结语

本设计稿将小程序定位为：**咨询订阅业务的移动侧延伸**，而非独立 CMS。后续 UI 稿可在「首页价值主张、我的订阅页、会员权益页」三处优先出高保真，再铺到发现与文章列表。

如需同步输出 **PRD 级用户故事（User Story）+ 验收标准**，可在本稿基础上拆迭代 backlog。
